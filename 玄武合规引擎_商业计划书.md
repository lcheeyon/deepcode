## 1. 执行摘要

**玄武合规引擎**是一款全自主、AI 驱动的企业级安全合规平台，能够同时扫描**源代码仓库**与**云基础设施配置**，对照多项安全治理政策框架进行深度比对分析，最终生成专业级审计 PDF 报告并附带修复建议。

### 时代背景：Mythos 模型的警示

2026年4月，Anthropic 公开披露其 Claude Mythos 模型的能力：该模型能够**自主发现并利用软件漏洞**，规模与深度远超任何先前工具——无论人工还是自动化手段。Mythos 在所有主流操作系统和浏览器中发现了数千个零日漏洞，其中最老的已有 27 年历史。超过 99% 的漏洞在披露时仍未修补。

Anthropic 因网络安全担忧拒绝公开发布 Mythos，并启动了 **Project Glasswing**——一个联合 Amazon、Apple、Google、Microsoft 等十二家科技巨头、承诺投入 1 亿美元的防御性修复联盟。

**这是继 Log4Shell 以来企业安全格局最深刻的转变**——但 Glasswing 的扫描范围仅限于开源基础设施，不涵盖：
- 企业专有应用代码
- 云基础设施配置
- AI 大模型集成应用

这正是玄武合规引擎所填补的空白。

### 核心价值主张

- **数据主权保障**：全私有化部署，代码永不离开企业安全边界
- **源代码 + 云配置一体化扫描**：单一报告覆盖三个合规层次
- **AI 深度推理**：基于大模型推理而非规则匹配，发现传统工具无法检测的漏洞
- **国产大模型支持**：原生支持通义千问 Qwen3-Coder、DeepSeek-R1，满足信创与数据主权要求
- **覆盖中国核心法规**：等保 2.0、《网络安全法》、《数据安全法》、《个人信息保护法》、《生成式 AI 管理办法》
- **多云支持**：AWS、阿里云、腾讯云、华为云、Azure、GCP，以及离线私有数据中心

## 2. 神话溯源：玄武引擎命名哲学

### 2.1 玄武：深海之守护神

**玄武**是中国传统神话中的北方守护神，四象之一，黑色龟蛇合体，象征北方、水、冬季与无上防护之力。

- **玄（xuán）**：深邃、神秘、幽暗、渊博——对应"Deep"（深度扫描）
- **武（wǔ）**：武力、守护、战将——对应"Guard"（守卫防护）

**玄武**因此成为"DeepGuard"在中文神话体系中最精确的映射：一位深藏玄机、以守护为本的神祇。

据道教典籍记载，玄武本为凡人，奉天命修炼于武当山。神明告知他：欲得真正洁净，须将吸纳世间污浊的肠胃舍弃。他将五脏投入海中，化为龟蛇二灵，自身则升华为北方玄武大帝。

**这一神话与企业合规的现实高度契合**：企业必须舍弃被历史漏洞污染的旧有工具（陈旧的规则扫描器、人工审计流程），以玄武引擎的 AI 深度推理取而代之，方能实现真正的安全洁净。

## 3. 问题陈述：企业合规的三重危机

### 3.1 中国合规监管的急剧收紧

中国及大中华区企业面临前所未有的合规压力，多部重量级法规相继生效并同步执行：

**《网络安全法》（2017年）**：强制要求关键信息基础设施运营者进行等保评估，违规最高罚款 100 万元并吊销营业执照。

**《数据安全法》（2021年）**：建立数据分类分级制度，重要数据处理须经安全评估，违规最高罚款 1,000 万元。

**《个人信息保护法》（2021年）**：类似欧盟 GDPR，对个人信息处理活动施加严格限制，违规最高罚款 5,000 万元或上年营业额 5%。

**网络安全等级保护 2.0（等保 2.0）**：GB/T 22239-2019 强制要求二级及以上系统每年开展等保测评，覆盖技术要求与管理要求共计数百条控制项。

**《生成式人工智能服务管理暂行办法》（2023年）**：国内首部专门针对生成式 AI 的监管规定，要求安全评估、内容过滤、数据标注规范，以及可追溯的风险管理体系。

**现实困境**：上述合规工作当前完全依赖人工审计，每次等保三级评估耗时 3–6 个月，费用 50–200 万元，且结论难以持续更新。没有任何现有工具能够将代码扫描与云配置检查一体化，并以 AI 推理取代规则匹配。

### 3.2 数据主权的绝对红线

中国法规对数据出境有严格限制：《数据安全法》第 31 条、《个人信息保护法》第 38 条均规定，重要数据和个人信息在出境前须经国家网信办安全评估。

**这意味着：企业代码与云配置数据绝对不能发送至境外 AI API（包括 Anthropic、OpenAI、Google）进行分析。**

玄武合规引擎的私有化部署模式（阿里云 VPC / 腾讯云 VPC / 华为云 VPC / 企业自有数据中心）是唯一合法且合规的解决路径：代码永不离开企业安全边界。

### 3.3 AI 漏洞发现能力的跃级威胁

Mythos 模型的披露证明了一个令人震惊的现实：**每一个企业代码库，都几乎确定含有传统扫描工具从未发现、也永远无法发现的深层漏洞。**

对中国企业而言，这一威胁尤为紧迫：
- 众多国内企业的核心业务系统建立于 10–20 年前的技术栈之上
- 等保 2.0 的检测能力基于规则库，无法检测 AI 可发现的新型深层漏洞类别
- 国家级威胁行为者（APT 组织）具有最强动机在 Mythos 类工具公开之前率先利用这些漏洞

### 3.4 现有工具的根本局限

| 工具类型 | 代表产品 | 核心局限 |
|---|---|---|
| 国内等保合规工具 | 天融信、绿盟、安恒 | 规则检测，无 AI 推理；不扫描源代码；不支持 AI 治理框架 |
| 国内 SAST 工具 | 奇安信代码卫士、默安科技 | 代码扫描，但无云配置关联；无自定义策略；无 AI 治理 |
| 国际 SAST 工具 | Snyk、Checkmarx、Veracode | 数据主权问题；不支持国产云；不支持中国法规框架 |
| 云原生安全工具 | 阿里云云安全中心、腾讯云安全 | 单云范围；不扫描源代码；无 LLM 推理；无自定义策略 |

## 4. 解决方案：玄武合规引擎

### 玄武合规引擎概述

一套全自主 AI 代理系统，接受以下任意组合输入：

- 源代码仓库（GitHub、GitLab、Gitee、Bitbucket 或 ZIP 上传）
- 云基础设施配置（IaC：Terraform、CloudFormation、CDK；国内：阿里云 ROS、华为云 CFW）
- 实时云环境连接（只读 API 扫描运行中的配置）
- 安全政策文档（预设框架或自定义上传）

输出一份专业合规报告，包含：

- 代码层、IaC 层、云配置层三层统一合规发现
- 每项控制要求的通过/失败/部分通过状态及精确证据引用
- 按漏洞可利用性排序的修复手册（代码补丁 + IaC 修复 + CLI 命令）
- 管理层摘要：总体合规得分与风险评级
- 适用于外部审计机构和监管机构的审计证据报告

### 4.1 部署模式

| 模式 | 描述 | 适用场景 |
|---|---|---|
| **SaaS（托管云）** | DeepGuard 多租户托管 | 中小企业、初创公司 |
| **私有云 — AWS** | 部署于客户 AWS VPC | 合规要求较高的跨国企业 |
| **私有云 — 阿里云** | 部署于客户阿里云 VPC | 国内企业、金融机构 |
| **私有云 — 腾讯云** | 部署于客户腾讯云 VPC | 互联网企业、游戏公司 |
| **私有云 — 华为云** | 部署于客户华为云 VPC | 政务、央企、军工配套 |
| **离线 / 私有数据中心** | Kubernetes + 开源大模型 | 国防、涉密、信创环境 |

### 4.2 扫描层次

**第一层 — 应用代码**：源代码仓库、依赖库、API、认证流程、数据处理逻辑、大模型集成模式

**第二层 — 基础设施即代码（IaC）**：Terraform、CloudFormation、阿里云 ROS、华为云模板——部署前政策合规检查

**第三层 — 实时云配置**：通过只读 API 扫描运行中的云资源——IAM 策略、网络拓扑、存储配置、加密设置、日志审计状态

### 4.3 三大国内云专属部署架构

玄武合规引擎为阿里云、腾讯云、华为云分别提供经过优化的原生部署架构，每套架构均使用该云厂商的原生托管服务，确保最佳性能、合规性与成本效益。

### 11.1 架构哲学：数据主权优先，多云原生

玄武合规引擎从设计之初即以**数据主权**为第一原则：

1. **每一处云基础设施边界均设抽象层**：存储、计算、密钥、队列、LLM 均通过云中立接口与可插拔适配器访问

3. **开放标准持久化**：PostgreSQL（pgvector）用于向量存储和状态；S3 兼容对象存储（阿里云 OSS、腾讯云 COS、华为云 OBS、MinIO 离线）；标准 OIDC 认证

## 12. 开源大语言模型选择

### 12.1 为何国产大模型是核心战略选择

玄武合规引擎将国产开源大模型置于战略核心，原因有三：

**合规性**：中国《数据安全法》和《个人信息保护法》要求重要数据不出境。支持通义千问 Qwen3-Coder、DeepSeek-R1 在客户自有环境中部署，是满足数据主权要求的唯一路径。

**性能**：国产代码大模型已达到并超越境外模型在代码任务上的能力基准：
- Qwen3-Coder-Next-80B：SWE-Bench Verified 超 70%，超越 GPT-4o
- DeepSeek-R1 671B：多步推理达到前沿水平，代码理解深度无可比拟

**政策加分**：信创评审中优先采用国产技术，使用国产大模型是进入信创目录的必要条件之一。

### 12.2 推荐模型（按部署场景）

#### 最高精度推理（大型 GPU 集群）

**DeepSeek-R1（671B 参数，MoE 架构，MIT 许可）**
- 最适用于：复杂跨文件漏洞推理、多步合规链分析
- 每次前向传播仅激活 37B 参数（MoE 效率）
- 硬件要求：4× NVIDIA A100 80GB 或同等设备
- 部署：vLLM 或 Ollama `deepseek-r1:671b`

**Qwen3-Coder-480B-A35B（通义千问，Apache 2.0）**
- SWE-Bench Verified 69.6%，超越 GPT-4o 和 Claude 3.7 Sonnet
- 上下文窗口：256K（可扩展至 1M），可摄取大型企业仓库单次完整分析
- 硬件要求：250GB 统一内存（4× H100 80GB）

#### 标准私有云部署（单 GPU 节点）

**Qwen3-Coder-Next-80B-A3B（通义千问，Apache 2.0）**
- SWE-Bench Verified 超 70%，仅激活 3B 参数——性价比极高
- 硬件要求：46GB VRAM，可在单张 H100 80GB 上运行
- **推荐为大多数私有云部署的默认选型**

**Qwen3-Coder-Plus（32B，Apache 2.0）**
- 最适用于标准单 GPU 节点部署
- 上下文 128K，1× A100 80GB

#### 资源受限 / 信创环境

**DeepSeek Coder V2（16B / 236B，许可证商业友好）**
- 支持 300+ 编程语言；适合多语言企业代码库
- 16B 版可在单张 A10G 24GB 运行

**Mixtral 8x7B（Apache 2.0）**
- 最低硬件门槛，适合现有 GPU 基础设施

## 13. 三大国内云基础设施扫描覆盖

玄武合规引擎通过各云平台只读 API 凭证扫描实时运行配置，每个云平台均有专属适配器，将原生 API 数据规范化为统一的**标准资源快照（ResourceSnapshot）**后送入合规推理引擎。扫描过程不需要部署代理，不写入任何数据，对生产环境零影响。

### 13.1 阿里云（Alibaba Cloud）扫描覆盖

通过只读 RAM 角色（RAM Role with STS Token）接入，扫描以下资源类型：

**身份与访问管理（RAM）**
- RAM 用户、角色、策略（内联和托管）——最小权限分析、通配符权限、权限提升路径
- 多因素认证（MFA）执行状态
- 服务控制策略（SCP）——组织级别防护栏

**网络安全**
- VPC 拓扑——子网分隔、流量日志、默认 VPC 使用
- 安全组——过度宽松的入站/出站规则（0.0.0.0/0 暴露）
- 网络 ACL、VPC 对等连接配置、PrivateLink 端点

**存储与数据**
- OSS 存储桶——公共访问屏蔽、ACL、存储桶策略、版本控制、服务端加密（SSE-OSS vs SSE-KMS）
- RDS 实例——加密、公开可访问性、备份保留、多可用区
- TableStore、KMS 密钥、证书管理

**计算与容器**
- ECS 实例——公网 IP 暴露、安全组、过期镜像
- ACK 集群——API Server 访问、加密、RBAC、节点安全策略
- 函数计算——函数策略、环境变量密钥、VPC 配置

**安全服务配置**
- 云安全中心——开启状态、基线检查规则覆盖、漏洞扫描配置
- ActionTrail 审计日志——多地域、日志完整性验证、OSS 投递配置
- 云盾 WAF——规则集、防护模式（观察/拦截）、IP 黑名单
- DDoS 防护——高防实例绑定、防护带宽配置
- 堡垒机（跳板机）——运维审计启用状态，等保三级强制要求

### 13.2 腾讯云（Tencent Cloud）扫描覆盖

通过只读子账号（CAM 策略：`ReadOnlyAccess`）接入，扫描以下资源类型：

**访问管理（CAM）**
- 子账号、协作者、角色——最小权限分析、通配符操作（`*`）检测、角色信任策略审查
- MFA 绑定状态——根账号及高权限子账号 MFA 强制执行检查
- API 密钥安全——密钥轮换状态、最后使用时间、过期策略
- 策略模拟器结果——权限实际生效路径分析

**网络安全**
- VPC 私有网络——子网划分、流量镜像、默认 VPC 隔离状态
- 安全组规则——0.0.0.0/0 入站暴露、端口范围过宽、出站限制缺失
- NAT 网关——EIP 绑定、出站路由策略
- 私有连接 / 专线接入（DC）——加密传输状态
- 网络访问控制（NACL）配置合规性

**存储与数据**
- 对象存储 COS——公有读写权限、存储桶策略、跨账号访问、版本控制、SSE-KMS 加密
- 数据库 TencentDB for MySQL / TDSQL-C——公网开放检测、数据加密、自动备份、多可用区
- Redis（云数据库 Redis）——认证密码、公网暴露、持久化配置
- 云加密机（CloudHSM）——密钥策略合规性

**计算与容器**
- 云服务器 CVM——公网 IP 暴露、安全组配置、镜像版本（过期内核）、启动模板安全
- 容器服务 TKE——API Server 公网访问控制、RBAC 策略、镜像仓库 TCR 扫描
- 云函数 SCF——触发器配置、环境变量明文密钥检测、VPC 接入状态
- 弹性伸缩（AS）——启动配置安全基线

**安全服务配置**
- 主机安全 CWP（云镜）——代理安装覆盖率、基线检查、漏洞修复状态
- 云审计 CloudAudit——操作日志投递 COS/CLS、日志完整性验证
- Web 应用防火墙 WAF——防护域名覆盖、规则模式
- 云防火墙 CFW——互联网边界防护、内网访问控制、IPS 功能状态
- 日志服务 CLS——日志组保留策略、加密配置

**等保专项（腾讯云等保合规）**
- 腾讯云等保合规评估服务状态（TCCC）
- 安全评级基线结果与差距分析

### 13.3 华为云（Huawei Cloud）扫描覆盖

**统一身份认证（IAM）**

- 登录保护——MFA 验证策略、登录 IP 白名单

**存储与数据**

**计算与容器**

- 云容器引擎 CCE——API Server 访问策略、RBAC 配置、节点池安全组、镜像签名（支持国密证书）
- 函数工作流 FunctionGraph——触发器权限、环境变量安全、VPC 配置

**安全服务配置**

- 主机安全服务 HSS——代理部署覆盖率、勒索病毒防护、漏洞扫描周期

| **等保 AI 系统专项** | 函数计算权限；ACK RBAC | SCF 触发器权限；TKE 镜像签名 | FunctionGraph VPC；CCE 镜像签名（国密） |
| **国密算法要求（信创）** | 部分支持 | 部分支持 | **SM2/SM3/SM4 全链路（强制项）** |

### 13.5 跨层关联——玄武的独特洞察

**跨层关联**是玄武合规引擎最强大的能力——发现基础设施承诺与应用代码实现之间的断层。以下为三云真实场景示例：

**阿里云场景**
- OSS 服务端加密已启用（基础设施通过）→ 应用代码将解密后的明文直接写入日志（代码失败）→ **加密被应用层绕过，等保 8.2.4 失败**
- RAM 角色已实施最小权限（IAM 通过）→ 函数计算代码在运行时动态假定更宽泛的角色（代码失败）→ **运行时权限提升，等保 8.1.3 失败**
- ActionTrail 已启用（配置通过）→ 应用全局捕获异常并静默处理，无任何日志输出（代码失败）→ **威胁检测被应用层致盲，等保 8.1.4 失败**

**腾讯云场景**
- TencentDB SSL 已强制（配置通过）→ 应用 JDBC 连接字符串设置 `useSSL=false`（代码失败）→ **传输中加密在代码层被禁用**
- CAM 策略已限制 COS 写入（IAM 通过）→ 应用通过临时密钥绕过 CAM 策略直接写入（代码失败）→ **临时凭证滥用，数据泄露风险**
- WAF 已开启（配置通过）→ 应用代码未对用户输入做 SQL 参数化处理（代码失败）→ **SQL 注入风险在应用层存在，WAF 旁路可利用**

**华为云场景**
- OBS 已配置 SM4 加密（配置通过）→ 应用代码使用明文 AK/SK 硬编码访问 OBS（代码失败）→ **密钥暴露风险，国密加密形同虚设**
- CCE 已配置镜像签名验证（配置通过）→ CI/CD 脚本跳过签名验证标志（IaC 失败）→ **供应链安全漏洞，信创等保强制要求违反**
- CBH 堡垒机已部署（配置通过）→ 应用运维代码包含直连数据库的后门连接字符串（代码失败）→ **堡垒机审计被绕过，等保 8.1.6 审计失效**

## 16. 报告输出结构

每份玄武合规报告包含：

1. **封面** — 企业名称、仓库/账户、扫描框架、扫描日期、报告密级、使用的 LLM 模型
2. **执行摘要** — 三层统一合规得分（代码/IaC/云配置）；风险态势评级；前 5 项关键发现；Mythos 时代漏洞面评估；即时整改要求
3. **合规态势仪表板** — 所有控制域与所有扫描层的可视化热力图
4. **跨层关联洞察** — 代码缺口与基础设施缺口相互叠加放大风险的发现
5. **发现清单** — 完整表格：控制要求编号、标题、严重性、状态、受影响资产（文件:行号 或 资源 ARN/ID）、CVSS 评分
6. **详细发现** — 每项发现：策略原文、证据（注释代码或资源配置）、LLM 推理链、严重性评级依据
7. **修复手册** — 按优先级排序：代码补丁（diff）、IaC 修复（Terraform/ROS/CFN）、CLI 命令、预估工作量、修复责任人
8. **基础设施配置摘要** — 每个云厂商：资源清单、类型覆盖率、态势得分
9. **方法论与局限性** — 分析方法、工具版本、LLM 模型及版本、置信度指标、已知局限
10. **附录 A：策略-控制映射** — 框架要求 → 发现 → 证据 → 修复可追溯矩阵
11. **附录 B：等保测评证据包** — 等保 2.0 格式化输出，可直接提交给测评机构
12. **附录 C：扫描配置** — 仓库提交哈希、云账户 ID、框架版本、扫描参数
